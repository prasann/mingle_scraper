Story#254 : Replay attack against Pass authorisation token
Internal Systems Involved: Pass Manager
Created On: Aug 22, 12:02 UTC (Liandra)
Last Modified by: Aug 27, 08:27 UTC (David Hughes)
Played in Iteration :

Attack Vector

Replay attack against Pass authorisation token

Sub Systems

iOS Passbook or Attido Mobile’s PassWallet. Pass Manager pass
update endpoint

Description

The authorization token is designed to be reusable as the iOS Passes can be synched or explicitly sent between devices. If a particular token is
compromised, an attacker can at any point retrieve the latest version of that particular Pass.

Notes

The authorization token is a randomly generated 30-character alpha-numeric string by Pass Manager . Tokens are not reused between
passes, even those belonging the same user. The authorization token is distributed to iOS devices as part of the Pass itself; if an attacker as gained hold of this secret then they very likely also have
the full Pass as well. The authorization header only allows them to retrieve later versions of the pass from Pass Manager.

Dave: I received the following sample code from Attido Mobile that can be embedded in the Android member app to allow it to directly transfer passes to PassWallet. 

The code to import a pass into PassWallet is as follows:

private static boolean launchPassWallet(Context applicationContext, Uri uri, boolean launchGooglePlay) {
    if (null != applicationContext) {
      PackageManager packageManager = applicationContext.getPackageManager();
      if (null != packageManager) {
        final String strPackageName = "com.attidomobile.passwallet";
        Intent passWalletLaunchIntent = packageManager.getLaunchIntentForPackage(strPackageName);
        Intent startIntent = new Intent();
        startIntent.setFlags(Intent.FLAG_ACTIVITY_NEW_TASK);
        startIntent.setAction(Intent.ACTION_VIEW);
        if (null == passWalletLaunchIntent) {
          // PassWallet isn't installed, open Google Play:
          if (launchGooglePlay) {
            String strReferrer = "";
            try {
              final String strEncodedURL = URLEncoder.encode(uri.toString(), "UTF-8");
              strReferrer = "&amp;referrer=" + strEncodedURL;
            } catch (UnsupportedEncodingException e) {
              e.printStackTrace();
              strReferrer = "";
            }
            try {
              startIntent.setData(Uri.parse("market://details?id=" + strPackageName + strReferrer));
              applicationContext.startActivity(startIntent);
            } catch (android.content.ActivityNotFoundException anfe) {
              startIntent.setData(Uri.parse("http://play.google.com/store/apps/details?id=" + strPackageName + strReferrer));
              applicationContext.startActivity(startIntent);
            }
          }
        } else {
          final String strClassName = "com.attidomobile.passwallet.activity.TicketDetailActivity";
          startIntent.setClassName(strPackageName, strClassName);
          startIntent.addCategory(Intent.CATEGORY_BROWSABLE);
          startIntent.setDataAndType(uri, "application/vnd.apple.pkpass");
          applicationContext.startActivity(startIntent);
          return true;
        }
      }
    }
    return false;
  }

Story#129 : general exception handling
Internal Systems Involved: Pass Manager
Created On: Jun 6, 08:48 UTC (David Hughes)
Last Modified by: Aug 2, 12:28 UTC (Uchenna)
Played in Iteration :4

h1. Narrative

Define error handling stategy and introduce it in the application by implementing strategy for token processing service.

h1. Tech Tasks

- Create error queue for pass manger application

- Implement error handling strategy for token processing service.

h1. Notes

Two types of services are exposed by “Pass Manager”, for which separate exception handling strategy need to be implemented.

Synchronous Services (ex : Download Pass, List Passes For Entity etc)

In case of any error during  any Synchronous Service Execution,  strategy should be to reply back immediately with proper error message, so that consumer can take necessary action (like retry or  data fixing etc). And this strategy need to be implemented  during implementation of such service.

 Asynchronous Services (ex : Token Processing (Create+Update), Update on entity change notification etc)

But in case of any error during any Asynchronous Service execution, strategy should be different for different types of errors.

There are 2 types of error that could occur  during execution of any asynchronous service execution.

-Technical Error (Should Retry)

Technical errors occurred in case of any technical failure like database connection failure , entity service connection failure etc. So in case of technical error “Pass Manager” should retry for a definite number of times to pass through the error step, expecting the technical error will be resolved in the next retry. And after the max retry limit reach it should be persisted in some managed environment like (Tibco CLE) so that it can be referred and processed manually later, if required. 

-Business Error (Should Not Retry)

Business error occurred in case of any business failure like invalid request format, invalid data or cases in which case retry is worthless. So in case of business error “Pass Manager” should generate an error message (source info+ initial request payload+ error type) and put it into Pass Manager’s error queue , so that it can be consumed by respective consumers or interested parties and  can take necessary action over the fault, which causes the error.
